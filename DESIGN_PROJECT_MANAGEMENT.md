# 项目管理模块 — 设计文档

> **定位**：这不是另一个 Jira/Linear/Notion。这是为**日志记录者**设计的项目管理，核心洞察是：**项目不是被管理的，而是从日常记录中自然涌现的**。

---

## 1. 设计哲学

### 1.1 核心理念：自下而上的项目涌现

传统项目管理工具假设你**先规划项目，再拆解任务**。但对于知识工作者、创作者、研究者，真实的工作流是：

```
日常碎片记录 → AI 识别模式 → 建议项目 → 用户确认 → 项目自然成型
```

**我们的差异化**：
- **Linear/Jira**：自上而下，项目经理分配任务
- **Notion**：空白画布，用户自己搭建
- **我们**：自下而上，AI 从日志中提炼项目，用户确认和塑造

### 1.2 三个设计原则

**原则 1：轻量优先（Lightweight by Default）**
- 项目不是官僚流程，不需要填 20 个字段
- 最小可用项目：标题 + 目标日期（2 个字段）
- 其他字段（描述、里程碑、干系人）按需补充

**原则 2：日志驱动（Journal-Driven）**
- 项目的"真相"不在项目卡片上，而在日常记录里
- 项目进度 = 关联日志的聚合，而非手动更新百分比
- AI 自动从日志中提取项目进展信号

**原则 3：反思闭环（Reflection Loop）**
- 项目不只是"完成"，而是"回顾"
- 每个完成的项目自动生成回顾报告（学到了什么、花了多久、哪些意外）
- 项目回顾成为个人/职业成长的素材

### 1.3 Solo vs Wolo 的设计差异

| 维度 | Solo（个人项目） | Wolo（工作项目） |
|------|------------------|------------------|
| **项目来源** | 个人目标、习惯、兴趣 | 工作交付物、OKR、团队协作 |
| **核心字段** | 目标、动机、习惯追踪 | 交付物、干系人、成功标准 |
| **进度信号** | 日志中的情绪、坚持天数 | 日志中的决策、阻塞、里程碑 |
| **完成标准** | 模糊（"感觉达成了"） | 明确（交付物验收） |
| **回顾重点** | 个人成长、习惯养成 | 复盘决策、积累经验 |
| **AI 角色** | 识别个人目标模式 | 识别工作阻塞和风险 |

## 2. 用户故事

### 2.1 个人项目场景（Solo）

**故事 1：从零散记录到项目**

> 小明每天用 solo 记录 3-5 条碎片。某天他回看过去两周的记录，发现提到了 8 次"想学 Rust"、3 次"看了 Rust 教程"。
>
> AI 主动提示："你最近多次提到学习 Rust，要不要创建一个'学习 Rust'项目？我可以帮你整理已有的相关记录。"
>
> 小明说"好"。AI 自动：
> 1. 创建项目"学习 Rust"
> 2. 将 8 条相关记录关联到项目
> 3. 建议 3 个里程碑："读完官方教程"、"写第一个 CLI 工具"、"贡献一个开源 PR"
> 4. 设置目标日期：3 个月后
>
> 小明微调了里程碑标题，确认。项目创建完成。

**故事 2：日常记录自动关联项目**

> 第二天，小明记录："今天花了 2 小时看 Rust ownership 章节，终于理解了借用检查器。"
>
> AI 自动识别：这条记录与"学习 Rust"项目相关，将其关联，并在项目时间线中添加一条活动。
>
> 周末，小明查看"学习 Rust"项目页面，看到：
> - 本周新增 3 条相关记录
> - 里程碑"读完官方教程"进度 60%（基于关联记录数量）
> - 时间线视图：每天的学习活动分布

**故事 3：项目回顾**

> 3 个月后，小明完成了所有里程碑。他点击"完成项目"。
>
> 系统自动生成回顾报告：
> - **总耗时**：87 天，42 条相关记录
> - **学习曲线**：前 2 周高频（兴奋期），第 3-4 周低谷（困难期），之后稳定
> - **关键突破**：第 18 天"理解借用检查器"、第 45 天"写出第一个异步程序"
> - **情绪趋势**：从焦虑 → 平静 → 兴奋
> - **建议**：你在第 3 周差点放弃，下次遇到类似低谷时，可以设置更小的里程碑
>
> 小明觉得这份回顾比项目本身更有价值。

### 2.2 工作项目场景（Wolo）

**故事 4：工作项目的自然识别**

> 小红是产品经理，每天用 wolo 记录工作碎片。过去一周她记录了：
> - "和工程师讨论新版搜索架构"
> - "用户反馈搜索结果不准"
> - "老板问搜索优化什么时候能上"
> - "写了搜索优化的 PRD 草稿"
>
> AI 提示："你最近在推进'搜索优化'相关的工作，要创建一个项目跟踪吗？"
>
> 小红确认。AI 创建项目并自动：
> - 关联 4 条记录
> - 建议里程碑："PRD 评审通过"、"技术方案确认"、"开发完成"、"上线验收"
> - 标记干系人："工程师"、"老板"
> - 识别风险："PRD 还在草稿阶段，需要尽快评审"

**故事 5：项目看板与阻塞管理**

> 小红在 onboard Web UI 中查看工作项目看板：
> - **Active（3 个）**：搜索优化（进度 40%，有阻塞标记）、用户增长实验（进度 70%）、Q4 OKR 规划（进度 20%）
> - **Completed（2 个）**：新人培训、竞品分析
> - **Archived（1 个）**：旧版文档迁移
>
> 她点击"搜索优化"，看到：
> - **阻塞项**："等待工程师给出技术可行性评估"（已阻塞 3 天）
> - **最近活动**：昨天更新了 PRD，3 天前和工程师开会
> - **决策记录**："决定先用 BM25 而非向量搜索"（附理由）
>
> 小红在项目详情页添加一条新记录："今天催了工程师，承诺周五前给评估。"系统自动更新阻塞状态为"跟进中"。

**故事 6：跨项目关联与周报生成**

> 周五下午，小红要写周报。她在 onboard 中点击"生成周报"。
>
> AI 自动汇总本周所有工作项目：
> - **搜索优化**：PRD 定稿，技术评估跟进中（阻塞）
> - **用户增长实验**：A/B 测试上线，等待数据
> - **Q4 OKR 规划**：完成草稿，等待老板反馈
>
> 周报还包含：
> - 本周决策："搜索优化选择 BM25 方案"
> - 本周阻塞："搜索优化等待技术评估（3 天）"
> - 下周重点："推动搜索优化技术评审、分析增长实验数据"
>
> 小红微调后发给老板，节省了 1 小时整理时间。

### 2.3 AI 辅助场景

**故事 7：AI 主动建议项目归档**

> 小明的"学习 Rust"项目已经 2 个月没有新记录。AI 提示：
>
> "你的'学习 Rust'项目最近 60 天没有活动，要归档它吗？或者你想重新激活？"
>
> 小明选择归档。系统记录归档原因（时间不足），并在年度报告中标记："学习 Rust 项目暂停，累计投入 42 小时"。

**故事 8：AI 识别项目风险**

> 小红的"搜索优化"项目里程碑"技术评审"原定本周完成，但已经延期 5 天。AI 提示：
>
> "里程碑'技术评审'已延期 5 天，检测到阻塞原因是'工程师资源紧张'。建议：1) 调整目标日期到下周三；2) 添加备选方案'如果评审继续延期，先用简化版上线'。"
>
> 小红采纳建议 1，项目时间线自动更新。

## 4. 管理方式

### 4.1 项目生命周期

```
创建 → 活跃 → 完成/归档
  ↑              ↓
  └── 重新激活 ──┘
```

**状态定义**：
- **Active（活跃）**：正在进行，有近期活动
- **Completed（完成）**：所有里程碑达成，用户确认完成
- **Archived（归档）**：长期无活动，或用户主动归档
- **重新激活**：归档后可恢复为活跃状态

### 4.2 进度计算逻辑

**进度 = f(里程碑完成度, 关联记录数, 时间衰减)**

```python
def calculate_progress(project):
    # 基础进度：里程碑完成比例（权重 70%）
    milestone_progress = completed_milestones / total_milestones * 100
    
    # 活动信号：最近 30 天关联记录数（权重 20%）
    recent_records = count_records(project_id, days=30)
    activity_score = min(recent_records / 10, 1.0) * 100
    
    # 时间衰减：距离目标日期越近，进度压力越大（权重 10%）
    days_remaining = (target_date - today).days
    time_pressure = max(0, 100 - days_remaining) if days_remaining > 0 else 100
    
    # 综合进度
    progress = (
        milestone_progress * 0.7 +
        activity_score * 0.2 +
        time_pressure * 0.1
    )
    
    return min(progress, 100)
```

### 4.3 AI 介入时机

| 时机 | AI 行为 | 用户选择 |
|------|---------|----------|
| **记录时** | 检测记录与项目的语义相似度，建议关联 | 确认/拒绝/新建项目 |
| **项目停滞** | 30 天无活动，提示归档或重新激活 | 归档/激活/忽略 |
| **里程碑延期** | 检测延期原因，建议调整日期或添加备选方案 | 调整/添加/忽略 |
| **项目完成** | 自动生成回顾报告 | 保存/编辑/删除 |
| **周报/月报** | 汇总项目进展，识别阻塞和风险 | 编辑后发送 |
| **年度报告** | 统计全年项目完成情况，识别成长轨迹 | 纳入报告 |

### 6.5 开放问题（待讨论）

**Q1：项目与 Area 的关系？**
- Things 3 有 Area（长期责任领域，如"健康"、"职业"），Project 属于 Area
- 我们要不要引入 Area 概念？还是用 tags 代替？
- **建议**：v1 用 tags，v2 考虑 Area

**Q2：项目进度是自动计算还是手动更新？**
- 自动：基于里程碑完成度和关联记录数
- 手动：用户自己设置百分比
- **建议**：自动为主，允许手动覆盖

**Q3：里程碑是否必须有目标日期？**
- 必须有：便于时间管理，但增加创建门槛
- 可选：降低门槛，但可能导致拖延
- **建议**：可选，但 AI 会建议添加

**Q4：项目回顾是自动触发还是手动触发？**
- 自动：完成所有里程碑时自动弹出
- 手动：用户点击"生成回顾"按钮
- **建议**：自动提示，用户可选择跳过

**Q5：如何处理"项目"与"习惯"的区别？**
- 项目有明确终点（如"学习 Rust"）
- 习惯是持续行为（如"每天运动"）
- **建议**：v1 不区分，用 tags 标记；v2 引入"习惯追踪"模块

## 7. 设计原则总结

| 原则 | 含义 | 反例 |
|------|------|------|
| **轻量优先** | 最小可用项目只需 2 个字段 | Jira：创建 issue 要填 20 个字段 |
| **日志驱动** | 项目进度从记录中涌现，而非手动更新 | Notion：需要自己维护进度条 |
| **AI 辅助** | AI 识别项目、关联记录、生成回顾 | Linear：所有操作需手动 |
| **反思闭环** | 项目完成不是终点，回顾才是 | 大多数工具：完成即归档，无回顾 |
| **Solo/Wolo 分化** | 个人项目重感受，工作项目重交付 | 通用工具：一套模型服务所有场景 |

## 8. 成功指标

**用户行为指标**：
- 日均创建项目数 > 0.1（每 10 天至少 1 个新项目）
- 项目关联记录占比 > 60%（大部分记录能自动关联到项目）
- 项目回顾生成率 > 80%（完成的项目大部分生成回顾）

**用户满意度指标**：
- "项目回顾报告有用" 认同度 > 4/5
- "AI 自动关联准确" 认同度 > 3.5/5
- "比不用工具更能追踪目标" 认同度 > 4/5

**产品健康指标**：
- 活跃项目数 / 用户 > 2（平均每人至少 2 个进行中项目）
- 项目完成率 > 50%（不是所有项目都必须完成，但过半完成是健康信号）
- 回顾报告查看率 > 70%（生成的回顾报告大部分被查看）

**结语**：这个项目管理模块的核心价值不是"管理项目"，而是**帮助用户看见自己的目标和努力**。项目是容器，记录是血肉，AI 是纽带，回顾是升华。
