# 项目管理模块 — 设计文档

> **定位**：这不是另一个 Jira/Linear/Notion。这是为**日志记录者**设计的项目管理，核心洞察是：**项目不是被管理的，而是从日常记录中自然涌现的**。

---

## 1. 设计哲学

### 1.1 核心理念：自下而上的项目涌现

传统项目管理工具假设你**先规划项目，再拆解任务**。但对于知识工作者、创作者、研究者，真实的工作流是：

```
日常碎片记录 → AI 识别模式 → 建议项目 → 用户确认 → 项目自然成型
```

**我们的差异化**：
- **Linear/Jira**：自上而下，项目经理分配任务
- **Notion**：空白画布，用户自己搭建
- **我们**：自下而上，AI 从日志中提炼项目，用户确认和塑造

### 1.2 四个设计原则

**原则 1：轻量优先（Lightweight by Default）**
- 项目不是官僚流程，不需要填 20 个字段
- 最小可用项目：标题（1 个字段）
- 目标日期、描述、里程碑、干系人、成功标准都按需补充
- 创建门槛要低于写一条记录，否则用户会绕过项目系统

**原则 2：日志驱动（Journal-Driven）**
- 项目的"真相"不在项目卡片上，而在日常记录里
- 项目进展来自记录、待办、决策、阻塞、里程碑的聚合，而非手动维护百分比
- 自动关联必须可追溯、可撤销，不能只靠标题或标签模糊匹配
- AI 从日志中提取进展信号，但最终项目状态由用户确认

**原则 3：反思闭环（Reflection Loop）**
- 项目不只是"完成"，而是"回顾"
- 每个完成的项目自动生成回顾报告（学到了什么、花了多久、哪些意外）
- 项目回顾成为个人/职业成长的素材

**原则 4：AI 建议，不越权（Assistive, Not Authoritative）**
- AI 可以建议创建、关联、完成、归档项目，但不应静默做高影响操作
- 高置信关联可以自动发生，但必须在记录反馈里展示并允许撤销
- 低置信或会改变项目生命周期的动作必须进入确认流

### 1.3 设计判断：V1 先做"可信项目容器"

我的建议：V1 不要一上来做完整 AI 项目经理，而是先做**结构化项目容器 + 可信关联层**。理由是项目实体、历史数据迁移、记录关联和 UI 聚合是后续 AI 能力的地基；如果先追求自动创建和自动判断进度，错误成本会直接污染用户的长期工作记忆。

V1 边界：
- `Project` / `Milestone` 是一等实体；记录、待办、决策、高亮、实验通过关联层挂到项目上
- 兼容现有 `solo.category` 和 `wolo.project` 字符串字段，但不继续把它们当作唯一项目来源
- AI 先做建议和半自动关联，不自动完成、归档、删除项目
- "阻塞"、"延期"、"停滞"是风险标记，不是项目生命周期状态，避免状态爆炸
- Web UI 先做看板、列表、详情页；完整甘特图放到后续版本

### 1.4 Solo vs Wolo 的设计差异

| 维度 | Solo（个人项目） | Wolo（工作项目） |
|------|------------------|------------------|
| **项目来源** | 个人目标、习惯、兴趣 | 工作交付物、OKR、团队协作 |
| **核心字段** | 目标、动机、感受 | 交付物、干系人、成功标准 |
| **进展信号** | 日志中的情绪、关键突破、活跃天数 | 日志中的决策、阻塞、里程碑 |
| **完成标准** | 用户确认或里程碑达成 | 明确交付物或验收标准 |
| **回顾重点** | 个人成长、习惯养成 | 复盘决策、积累经验 |
| **AI 角色** | 识别个人目标模式 | 识别工作阻塞和风险 |

## 2. 用户故事

### 2.1 个人项目场景（Solo）

**故事 1：从零散记录到项目**

> 小明每天用 solo 记录 3-5 条碎片。某天他回看过去两周的记录，发现提到了 8 次"想学 Rust"、3 次"看了 Rust 教程"。
>
> AI 主动提示："你最近多次提到学习 Rust，要不要创建一个'学习 Rust'项目？我可以帮你整理已有的相关记录。"
>
> 小明说"好"。AI 生成项目草稿：
> 1. 项目标题："学习 Rust"
> 2. 候选关联：8 条相关记录
> 3. 建议里程碑："读完官方教程"、"写第一个 CLI 工具"、"贡献一个开源 PR"
> 4. 建议目标日期：3 个月后
>
> 小明微调了里程碑标题，确认。项目创建完成。

**故事 2：日常记录自动关联项目**

> 第二天，小明记录："今天花了 2 小时看 Rust ownership 章节，终于理解了借用检查器。"
>
> AI 高置信识别：这条记录与"学习 Rust"项目相关，自动关联并在反馈里提示"可撤销"。如果置信度不够，只提示用户确认，不静默关联。
>
> 周末，小明查看"学习 Rust"项目页面，看到：
> - 本周新增 3 条相关记录
> - 里程碑"读完官方教程"进度 60%（基于用户确认的里程碑状态和相关待办）
> - 时间线视图：每天的学习活动分布

**故事 3：项目回顾**

> 3 个月后，小明完成了所有里程碑。他点击"完成项目"。
>
> 系统自动生成回顾报告：
> - **项目周期**：87 天，42 条相关记录，38 个活跃日
> - **学习曲线**：前 2 周高频（兴奋期），第 3-4 周低谷（困难期），之后稳定
> - **关键突破**：第 18 天"理解借用检查器"、第 45 天"写出第一个异步程序"
> - **情绪趋势**：从焦虑 → 平静 → 兴奋
> - **建议**：你在第 3 周差点放弃，下次遇到类似低谷时，可以设置更小的里程碑
>
> 小明觉得这份回顾比项目本身更有价值。

### 2.2 工作项目场景（Wolo）

**故事 4：工作项目的自然识别**

> 小红是产品经理，每天用 wolo 记录工作碎片。过去一周她记录了：
> - "和工程师讨论新版搜索架构"
> - "用户反馈搜索结果不准"
> - "老板问搜索优化什么时候能上"
> - "写了搜索优化的 PRD 草稿"
>
> AI 提示："你最近在推进'搜索优化'相关的工作，要创建一个项目跟踪吗？"
>
> 小红确认。系统创建项目并：
> - 关联 4 条记录
> - 建议里程碑："PRD 评审通过"、"技术方案确认"、"开发完成"、"上线验收"
> - 标记干系人："工程师"、"老板"
> - 识别风险："PRD 还在草稿阶段，需要尽快评审"

**故事 5：项目看板与阻塞管理**

> 小红在 onboard Web UI 中查看工作项目看板：
> - **Active（3 个）**：搜索优化（进度 40%，有阻塞标记）、用户增长实验（进度 70%）、Q4 OKR 规划（进度 20%）
> - **Completed（2 个）**：新人培训、竞品分析
> - **Archived（1 个）**：旧版文档迁移
>
> 她点击"搜索优化"，看到：
> - **阻塞项**："等待工程师给出技术可行性评估"（已阻塞 3 天）
> - **最近活动**：昨天更新了 PRD，3 天前和工程师开会
> - **决策记录**："决定先用 BM25 而非向量搜索"（附理由）
>
> 小红在项目详情页添加一条新记录："今天催了工程师，承诺周五前给评估。"系统自动更新阻塞状态为"跟进中"。

**故事 6：跨项目关联与周报生成**

> 周五下午，小红要写周报。她在 onboard 中点击"生成周报"。
>
> AI 自动汇总本周所有工作项目：
> - **搜索优化**：PRD 定稿，技术评估跟进中（阻塞）
> - **用户增长实验**：A/B 测试上线，等待数据
> - **Q4 OKR 规划**：完成草稿，等待老板反馈
>
> 周报还包含：
> - 本周决策："搜索优化选择 BM25 方案"
> - 本周阻塞："搜索优化等待技术评估（3 天）"
> - 下周重点："推动搜索优化技术评审、分析增长实验数据"
>
> 小红微调后发给老板，节省了 1 小时整理时间。

### 2.3 AI 辅助场景

**故事 7：AI 主动建议项目归档**

> 小明的"学习 Rust"项目已经 2 个月没有新记录。AI 提示：
>
> "你的'学习 Rust'项目最近 60 天没有活动，要归档它吗？或者你想重新激活？"
>
> 小明选择归档。系统记录归档原因（时间不足），并在年度报告中标记："学习 Rust 项目暂停，累计 42 条相关记录、38 个活跃日"。

**故事 8：AI 识别项目风险**

> 小红的"搜索优化"项目里程碑"技术评审"原定本周完成，但已经延期 5 天。AI 提示：
>
> "里程碑'技术评审'已延期 5 天，检测到阻塞原因是'工程师资源紧张'。建议：1) 调整目标日期到下周三；2) 添加备选方案'如果评审继续延期，先用简化版上线'。"
>
> 小红采纳建议 1，项目时间线自动更新。

## 4. 管理方式

### 4.1 项目生命周期

```
创建 → 活跃 → 完成/归档
  ↑              ↓
  └── 重新激活 ──┘
```

**状态定义**：
- **Active（活跃）**：正在进行，有近期活动
- **Completed（完成）**：所有里程碑达成，用户确认完成
- **Archived（归档）**：长期无活动，或用户主动归档
- **重新激活**：归档后可恢复为活跃状态

### 4.2 进度、健康度与风险分离

核心修正：**临近截止日期不是进度，记录变多也不一定代表完成**。V1 应该把"完成度"、"活跃度"和"风险"拆开展示，避免一个百分比承载过多含义。

| 指标 | 含义 | 计算来源 | 展示方式 |
|------|------|----------|----------|
| **完成度** | 交付或目标完成了多少 | 里程碑完成率为主；无里程碑时用关联待办完成率；都没有时显示"未量化" | 进度条 |
| **活跃度** | 最近是否持续推进 | 最近 7/30 天相关记录、待办、决策、高亮数量 | 活动热度、最近活动时间 |
| **风险** | 是否可能延期或停滞 | 目标日期、未完成里程碑、阻塞高亮、无活动天数 | 徽标：正常 / 关注 / 有风险 |

推荐规则：
- 有里程碑：完成度 = 已完成里程碑 / 总里程碑
- 无里程碑但有关联待办：完成度 = 已完成待办 / 总待办
- 无里程碑且无待办：不显示百分比，只显示活动和下一步建议
- 目标日期只影响风险，不直接提高或降低完成度
- 用户可以手动标记项目完成，但不提供任意百分比编辑，避免数据口径失真

### 4.3 AI 介入时机

| 时机 | AI 行为 | 用户选择 |
|------|---------|----------|
| **记录时** | 检测记录与项目的语义相似度；高置信自动关联并提示可撤销，中置信请求确认 | 确认/拒绝/撤销/新建项目 |
| **项目停滞** | 30 天无活动，提示归档或重新激活 | 归档/激活/忽略 |
| **里程碑延期** | 检测延期原因，建议调整日期或添加备选方案 | 调整/添加/忽略 |
| **项目完成** | 自动生成回顾报告 | 保存/编辑/删除 |
| **周报/月报** | 汇总项目进展，识别阻塞和风险 | 编辑后发送 |
| **年度报告** | 统计全年项目完成情况，识别成长轨迹 | 纳入报告 |

**置信度策略**：
- **高置信**：项目名称、别名、明确上下文或近期连续记录都匹配；可自动关联，但必须展示关联结果和撤销入口
- **中置信**：只有主题相似或关键词接近；进入确认队列，不改变项目数据
- **低置信**：只作为搜索召回，不展示打扰用户
- **生命周期动作**（完成、归档、删除、目标日期大幅调整）永远需要用户确认

### 4.4 权限与协作（Wolo 专属）

**当前版本（v1）**：单人使用，无协作
- 项目归属个人 wolo workspace
- 干系人字段仅为记录，不涉及权限

**未来版本（v2）**：团队协作
- 项目可共享给团队成员
- 里程碑可分配给不同人
- 活动流显示谁做了什么

### 4.5 数据模型与兼容性

项目管理不能只靠 `tags`、`category` 或 `project` 字符串字段，否则重命名、同名项目、历史导入和跨实体聚合都会变脆。推荐引入稳定 ID 和统一关联层。

```
用户记录 / 待办 / 决策 / 高亮 / 实验
          │
          ▼
   project_links（可追溯、可撤销、带置信度）
          │
          ▼
 Project ───── Milestone
          │
          ▼
 看板 / 详情页 / 报告 / 回顾
```

核心实体：
- **Project**：`id`, `title`, `description`, `status`, `priority`, `start_date`, `target_date`, `tags`, `created_at`, `updated_at`
- **Milestone**：`id`, `project_id`, `title`, `status`, `target_date`, `completed_at`
- **ProjectLink**：`project_id`, `entity_type`, `entity_id`, `source`, `confidence`, `status`, `created_at`
- **ProjectAlias**：`project_id`, `alias`，用于兼容旧 `wolo.project`、`solo.category` 和用户常用简称

`ProjectLink.source` 建议取值：`user`, `ai_high_confidence`, `ai_candidate`, `migration`。`ProjectLink.status` 建议取值：`active`, `pending`, `rejected`。

兼容策略：
- `solo.category` 和 `wolo.project` 保留为展示和导入线索，但新功能优先使用 `project_id`
- 历史数据迁移时不强行改写所有记录，而是生成待确认的 `ProjectLink` 建议
- 删除项目默认只删除项目容器和关联关系，不删除原始记录、待办、决策和高亮
- 项目重命名只改 `Project.title`，不破坏历史关联

## 5. 使用方法

### 5.1 快速开始

**步骤 1：创建第一个项目**

```bash
# Solo 个人项目
solo project create "学习 Rust" --target-date 2026-09-30

# Wolo 工作项目
wolo project create "搜索优化" --target-date 2026-08-15 --stakeholders "工程师,老板"
```

**步骤 2：添加里程碑**

```bash
solo project milestone "学习 Rust" "读完官方教程" --target-date 2026-07-30
solo project milestone "学习 Rust" "写第一个 CLI 工具" --target-date 2026-08-31
```

**步骤 3：日常记录时关联项目**

```bash
# 高置信自动关联，低置信进入确认
solo record "今天花了 2 小时看 Rust ownership"
# → AI: 已关联项目「学习 Rust」，可撤销；里程碑进展等待确认
```

**步骤 4：查看项目进度**

```bash
# CLI 快速查看
solo project list

# Web UI 详细查看
solo onboard run
# 浏览器打开 http://localhost:8090/projects
```

**步骤 5：项目回顾**

```bash
# 完成项目后生成回顾报告
solo project review "学习 Rust"
# → 自动生成：项目周期、关键突破、情绪趋势、建议
```

### 5.2 高级用法

**批量关联历史记录**

```bash
# 生成过去 30 天与"Rust"相关的候选关联，确认后写入
solo project link "学习 Rust" --keyword "Rust" --days 30 --review
```

**项目模板**

```bash
# 保存当前项目为模板
solo project template "学习 Rust" --name "学习新语言"

# 从模板创建新项目
solo project create "学习 Go" --template "学习新语言"
```

**跨项目统计**

```bash
# 查看所有项目的活跃度分布
solo project stats --period 2026-Q2
# → 输出：学习 Rust 45%，写书 30%，健身 25%（按相关记录和活跃日估算）
```

### 5.3 集成场景

**与年度报告集成**

```bash
# 生成年度报告时自动包含项目总结
solo report yearly 2026
# → 报告中包含：
#   - 本年度完成的项目（3 个）
#   - 进行中项目（2 个）
#   - 项目活跃度分布图
#   - 项目回顾亮点摘录
```

**与周报集成**

```bash
# 生成周报时自动汇总项目进展
wolo report weekly
# → 报告中包含：
#   - 本周活跃项目（3 个）
#   - 本周完成的里程碑（1 个）
#   - 阻塞项（2 个）
#   - 下周重点建议
```

## 6. 范围切片与需求澄清

### 6.1 V1 MVP：先让项目可信地存在

- [ ] 项目 CRUD（创建、查看、更新、删除、完成、归档、重新激活）
- [ ] 里程碑管理（添加、编辑、完成、删除；目标日期可选）
- [ ] 稳定关联层：记录、待办、决策、高亮、实验可关联到项目
- [ ] 进度、活跃度、风险分离展示
- [ ] CLI 命令：`list`, `create`, `show`, `update`, `delete`, `archive`, `milestone`, `link`, `unlink`
- [ ] Web UI 看板视图（Active / Completed / Archived）
- [ ] Web UI 列表视图（适合筛选、排序、批量查看）
- [ ] Web UI 详情页（概览、里程碑、任务、活动、统计）
- [ ] 项目回顾报告手动生成，完成项目时自动提示

### 6.2 V1.1：AI 辅助，但保留确认权

- [ ] 记录时高置信自动关联并支持撤销
- [ ] 中置信候选关联进入确认队列
- [ ] 批量关联历史记录，默认 review 后写入
- [ ] 项目停滞提醒（默认 30 天无活动）
- [ ] 里程碑延期和阻塞建议
- [ ] 与周报/月报/年度报告集成

### 6.3 V2：增强视图与协作能力

- [ ] 轻量时间线视图；完整甘特图仅在真实用户需要时引入
- [ ] 项目模板
- [ ] 项目标签/分类筛选
- [ ] 项目优先级排序
- [ ] 项目统计图表（活跃趋势、完成趋势、项目分布）
- [ ] 导出项目数据（CSV/JSON/Markdown）
- [ ] 项目回顾分享（Markdown 或图片）
- [ ] 团队协作与权限管理（Wolo 专属）

### 6.4 明确不做（Won't Have）

- [ ] V1 不做多人协作
- [ ] V1 不做权限管理
- [ ] V1 不做项目依赖关系
- [ ] 不做精确时间追踪（不要求用户记录小时数）
- [ ] 不与外部工具同步（Jira/Asana/Linear）
- [ ] 不做移动端 App
- [ ] 不做 AI 静默创建、完成、归档、删除项目

### 6.5 已收敛的设计决策

**D1：项目与 Area 的关系**
- V1 用 tags 覆盖长期领域，如"健康"、"职业"、"学习"
- 不引入 Area 实体，避免在项目实体尚未验证前增加一层分类体系

**D2：项目进度口径**
- 自动计算完成度，但只基于里程碑和待办
- 记录数量只影响活跃度，不影响完成度
- 不提供任意百分比手动编辑；用户通过完成里程碑、待办或项目来改变进度

**D3：里程碑目标日期**
- 目标日期可选
- AI 可以建议补日期，但不能把日期作为创建门槛

**D4：项目回顾触发**
- 用户可随时手动生成
- 项目完成时自动提示生成，但允许跳过

**D5：项目与习惯的边界**
- 项目必须有可结束的目标，如"减脂 5kg"、"完成 Rust CLI"
- 持续习惯如"每天运动"不作为项目建模；V1 可用 tags 或实验记录承载

## 7. 设计原则总结

| 原则 | 含义 | 反例 |
|------|------|------|
| **轻量优先** | 最小可用项目只需标题 | Jira：创建 issue 要填 20 个字段 |
| **日志驱动** | 项目事实从记录和关联实体中涌现，而非靠项目卡片自说自话 | Notion：需要自己维护进度条 |
| **AI 辅助** | AI 识别项目、建议关联、生成回顾，但保留确认和撤销 | Linear：所有操作需手动 |
| **反思闭环** | 项目完成不是终点，回顾才是 | 大多数工具：完成即归档，无回顾 |
| **Solo/Wolo 分化** | 个人项目重感受，工作项目重交付 | 通用工具：一套模型服务所有场景 |

## 8. 成功指标

**激活指标**：
- 首个项目创建率 > 40%（使用 onboard 或 CLI 的用户中）
- 首个项目创建到首条关联记录的中位时间 < 10 分钟
- 活跃项目数 / 用户 > 2（平均每人至少 2 个进行中项目）

**关联质量指标**：
- AI 候选关联接受率 > 70%
- 自动关联撤销率 < 10%
- 项目关联的手动修正率 < 15%
- 低置信建议曝光率 < 20%（避免打扰）

**结果指标**：
- 项目完成率 > 40%（不是所有项目都必须完成，但需要能推动收束）
- 完成项目的回顾生成率 > 70%
- 回顾报告查看率 > 70%
- "项目回顾报告有用" 认同度 > 4/5

**反指标**：
- 如果用户频繁删除项目，说明 AI 建议创建过度或项目边界不清
- 如果项目长期只有记录没有里程碑/待办，说明项目容器没有提供管理价值
- 如果用户大量手动解除关联，说明语义匹配阈值过低或别名机制不足

---

**结语**：这个项目管理模块的核心价值不是"管理项目"，而是**帮助用户看见自己的目标和努力**。项目是容器，记录是血肉，AI 是纽带，回顾是升华。
